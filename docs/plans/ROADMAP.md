# PathLab Viewer — PathLab Forge Integration Roadmap

This roadmap covers only server work in `PathLab-Viewer`. Desktop development belongs in the separate `PathLab-Forge` repository.

## Milestone 1 — Contract foundation

Outcome:

- canonical `.plslide` v1 JSON Schema;
- typed Python manifest validation;
- deterministic tiny package fixtures generated during tests;
- stable manifest/schema error codes;
- no production behavior changes.

Tasks:

1. Contract, typed manifest and fixture generator.
2. Safe TAR and DZI validator.

## Milestone 2 — Local asynchronous import

Outcome:

- explicit `prepared_package` ingest mode;
- `queued_import` and `importing` lifecycle;
- database migration preserving existing slides;
- bounded staging and atomic canonical asset installation;
- local synthetic package reaches `ready_private` without WSI conversion.

## Milestone 3 — Resumable desktop upload API

Outcome:

- capability negotiation;
- scoped prepared-slide reservation;
- reuse of existing tus infrastructure;
- length and SHA-256 verification;
- asynchronous import after completion;
- status polling and private-preview URL;
- legacy browser upload remains functional.

Initial authentication may use a revocable scoped token. Browser device pairing is a later hardening task.

## Milestone 4 — Single-copy publication

Outcome:

- one canonical DZI tree per slide;
- private preview reads canonical assets;
- public access maps atomically to the same assets;
- unpublish, delete and restore are safe;
- tests prove publishing does not duplicate the complete tile tree.

## Milestone 5 — Desktop authorization and operations

Outcome:

- browser-assisted device pairing;
- scoped, revocable desktop credentials;
- device audit trail;
- backup/restore includes prepared slides and reconstructs public mappings;
- low-disk and interrupted-import recovery.

## Milestone 6 — Integration evidence

Outcome:

- actual package generated by PathLab Forge imports successfully;
- VSI, SVS and OME-TIFF desktop outputs preview and publish;
- interrupted upload resumes;
- malformed and malicious packages are rejected;
- existing legacy slides and uploads regress cleanly;
- external viewer, shaped-network, physical-device and restore evidence is recorded.

## Execution rule

Only one active task appears in `docs/plans/active/current.md`. Codex must complete, commit, report and stop before that file is advanced.
